Aprende a trabajar con Git y GitHub 100% desde la terminal

Lección 02

Configurando Git en la Terminal

Git ya viene instalado en la mayoría de distribuciones Linux. Lo único que necesitamos es configurarlo con nuestros datos.

La configuración de Git tiene tres niveles: system (todo el equipo), global (nuestro usuario) y local (un proyecto concreto). Nosotros configuraremos a nivel global para que aplique a todos nuestros proyectos.

Comandos

# Verificar la versión de Git
git --version
# git version 2.34.1

# Configurar nombre y email (los que usaremos en GitHub)
git config --global user.name "Nombre Apellido"
git config --global user.email "email@ejemplo.com"

# Configurar el editor por defecto
git config --global core.editor "nano"

# Configurar la rama por defecto como main
git config --global init.defaultBranch main

# Verificar toda la configuración
git config --global --list
Lección 03

Creando Cuenta en GitHub y Autenticación SSH

Vamos a crear una cuenta nueva en GitHub desde cero. Una vez creada, necesitamos una forma de autenticarnos desde la terminal para poder subir y bajar código. Hay dos formas principales: HTTPS con tokens y SSH con claves. Usaremos SSH porque es lo más cómodo en el día a día: se configura una vez y no hay que estar introduciendo credenciales nunca más.

SSH funciona con un par de claves: una privada (que se queda en nuestro equipo y nunca se comparte) y una pública (que le damos a GitHub). Cuando nos conectamos, GitHub verifica que nuestra clave privada corresponde con la pública que tiene registrada.

Pasos en GitHub

  1. Ir a github.com y crear una cuenta nueva
  2. Verificar el email

Comandos

# Verificar que no tenemos claves SSH (dará error, es normal en una instalación nueva)
ls -la ~/.ssh/
# ls: cannot access '/home/usuario/.ssh/': No such file or directory

# Generar par de claves SSH con nombre descriptivo (pulsar Enter para passphrase vacía)
# La passphrase es una contraseña adicional que protege la clave privada.
# Si la dejamos vacía, no nos la pedirá cada vez que hagamos push o pull.
# En un equipo personal es cómodo y seguro. En equipos compartidos, mejor ponerla.
ssh-keygen -t ed25519 -C "email@ejemplo.com" -f ~/.ssh/github

# Verificar que se han creado las claves
ls -la ~/.ssh/
# github (clave privada) y github.pub (clave pública)

# Copiar la clave pública para pegarla en GitHub
cat ~/.ssh/github.pub

Pasos en GitHub

  1. Ir a Settings > SSH and GPG keys > New SSH key
  2. Pegar la clave pública copiada
  3. Guardar

Configuración SSH

# Indicar a SSH qué clave usar para GitHub
# Crear el archivo de configuración SSH
nano ~/.ssh/config

Contenido de ~/.ssh/config:

Host github.com
    IdentityFile ~/.ssh/github

Verificación

# Verificar que la conexión funciona
ssh -T git@github.com

# Debería responder: Hi usuario! You've successfully authenticated
Lección 04

Crear Proyecto Laravel y Subirlo a GitHub

Vamos a crear nuestro proyecto Laravel con Sail, que será el proyecto sobre el que trabajaremos durante todo el curso. Después crearemos un repositorio en GitHub y lo conectaremos. Esto es algo que haremos constantemente en nuestro día a día: iniciar un proyecto y vincularlo con un repositorio remoto.

Importante: Antes de ejecutar los comandos, asegúrate de que Docker Desktop esté ejecutándose. Laravel Sail depende de Docker para funcionar.

Pasos en GitHub

  1. Crear un nuevo repositorio vacío (sin README, sin .gitignore, sin licencia)
  2. Nombre: git-pro-course
  3. Visibilidad: público

Comandos

# Asegúrate de que Docker Desktop esté corriendo antes de continuar

# Crear proyecto Laravel con Sail
curl -s "https://laravel.build/git-pro-course" | bash

# Entrar al proyecto
cd git-pro-course

# Levantar el proyecto con Sail
sail up -d

# Verificar que funciona
sail artisan --version

# Inicializar Git
git init

# Ver el estado
git status

# Añadir todos los archivos
git add .

# Primer commit
git commit -m "chore: initial Laravel project setup"

# Conectar con nuestro repositorio remoto
git remote add origin git@github.com:USUARIO/git-pro-course.git

# Subir el proyecto
git push -u origin main
Lección 05

Conventional Commits - Mensajes Profesionales

En el curso de iniciación aprendimos a hacer commits, pero los mensajes eran libres. En equipos profesionales se usa una convención llamada Conventional Commits que estandariza los mensajes. Esto facilita entender el historial, generar changelogs automáticos y trabajar con herramientas de versionado.

La estructura es: tipo(ámbito opcional): descripción

Tipos principales

  • feat: nueva funcionalidad
  • fix: corrección de un bug
  • docs: cambios en documentación
  • style: cambios de formato (espacios, comas, etc., no CSS)
  • refactor: cambio de código que no añade funcionalidad ni corrige bugs
  • chore: tareas de mantenimiento (dependencias, configuración)
  • test: añadir o modificar tests

Ejemplos reales

  • feat: add welcome message to home page
  • fix: correct typo in about page title
  • docs: update README with installation steps
  • chore: update laravel framework to v11

Comandos

# Crear una vista para nuestra home
sail artisan make:view home

Código

Editar resources/views/home.blade.php:

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Git Pro Course</title>
</head>
<body>
    <h1>Welcome to Git Pro Course</h1>
    <p>Learning Git like a professional.</p>
</body>
</html>

Editar routes/web.php:

<?php

use Illuminate\Support\Facades\Route;

Route::get('/', function () {
    return view('home');
});

Comandos

# Ver los cambios
git status
git diff

# Commit con Conventional Commits (aún no tenemos aliases, los crearemos en la siguiente lección)
git add resources/views/home.blade.php
git add routes/web.php
git commit -m "feat: add home page with welcome message"

Código

Editar README.md:

# Git Pro Course

Proyecto Laravel utilizado para aprender Git y GitHub de forma profesional.

## Requisitos

- Docker Desktop
- WSL 2

## Instalación

git clone git@github.com:USUARIO/git-pro-course.git
cd git-pro-course
sail up -d

Comandos

git add README.md
git commit -m "docs: add project description and installation steps"

# Subir los cambios
git push

# Verificar el historial limpio
git log --oneline
Lección 06

Aliases de Git - Trabaja Más Rápido

Hay comandos de Git que escribimos decenas de veces al día. Los aliases nos permiten crear atajos personalizados para no repetir lo mismo constantemente. Es una configuración personal, cada desarrollador tiene los suyos, y se guardan en la configuración global de Git.

Comandos

# Aliases esenciales
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.ci commit

# Commit con mensaje directo (sin abrir editor)
git config --global alias.cm "commit -m"

# Log visual con gráfico de ramas, una línea por commit
git config --global alias.lg "log --oneline --graph --all --decorate"

# Ver solo el último commit
git config --global alias.last "log -1 --oneline"

# Sacar archivos del staging area (deshacer git add)
git config --global alias.unstage "restore --staged"

# Ver diferencias de archivos que están en staging
git config --global alias.ds "diff --staged"

# Probar los aliases
git st
git lg
git last

# Ver todos los aliases configurados
git config --global --get-regexp alias

Código

Editar resources/views/home.blade.php, añadir un footer:

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Git Pro Course</title>
</head>
<body>
    <h1>Welcome to Git Pro Course</h1>
    <p>Learning Git like a professional.</p>
    <footer>
        <p>&copy; 2026 Git Pro Course</p>
    </footer>
</body>
</html>

Comandos

# Usar aliases para el flujo completo
git st
git add .

# Ver los cambios que van a ir en el commit (ya staged)
git ds

git cm "feat: add footer to home page"
git lg
git push
Lección 07

Git Stash - Guardando Trabajo Temporal

Estamos trabajando en algo y de repente nos piden que revisemos otra cosa urgente en otra rama. Pero no queremos hacer commit de un trabajo a medias. Aquí es donde entra git stash: guarda nuestros cambios temporalmente en una pila, dejando el directorio de trabajo limpio para poder cambiar de rama. Cuando volvamos, recuperamos los cambios y seguimos donde lo dejamos.

Podemos tener múltiples stashes apilados, ponerles nombre para identificarlos, y aplicarlos selectivamente.

Código: trabajo en progreso

Editar resources/views/home.blade.php:

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Git Pro Course</title>
</head>
<body>
    <nav>
        <a href="/">Home</a>
        <a href="/about">About</a>
    </nav>
    <h1>Welcome to Git Pro Course</h1>
    <p>Learning Git like a professional.</p>
    <footer>
        <p>&copy; 2026 Git Pro Course</p>
    </footer>
</body>
</html>

Comandos

# Tenemos cambios sin terminar
git st

# Guardar en stash con un mensaje descriptivo
git stash push -m "wip: navigation menu"

# El directorio está limpio
git st

# Ver la lista de stashes
git stash list

# Vamos a atender una tarea urgente en otra rama
git co -b fix/add-about-page

Código: tarea urgente

Crear resources/views/about.blade.php:

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>About - Git Pro Course</title>
</head>
<body>
    <h1>About</h1>
    <p>This project is built with Laravel and Sail.</p>
</body>
</html>

Añadir ruta en routes/web.php:

<?php

use Illuminate\Support\Facades\Route;

Route::get('/', function () {
    return view('home');
});

Route::get('/about', function () {
    return view('about');
});

Comandos

# Commit de la tarea urgente
git add .
git cm "feat: add about page"

# Integrar la tarea urgente en main
git co main
git merge fix/add-about-page
git push
git br -d fix/add-about-page

# Ahora recuperamos nuestro trabajo del stash
git stash list
git stash pop

# Verificar que nuestros cambios del nav están de vuelta
git st
git diff

# Terminar el trabajo y hacer commit
git add .
git cm "feat: add navigation menu to home page"
git push

# Ver todos los stashes con su índice
# git stash list
# stash@{0} es siempre el más reciente, {1} el siguiente, etc.

# Aplicar un stash concreto sin eliminarlo de la lista
# git stash apply stash@{1}

# Eliminar un stash concreto sin aplicarlo
# git stash drop stash@{1}

# Limpiar todos los stashes
# git stash clear
Lección 08

Git Rebase - Reescribiendo la Historia

Merge crea un commit adicional de fusión, lo cual ensucia el historial. Rebase es la alternativa: toma los commits de nuestra rama y los "reaplica" encima de la rama destino, creando un historial lineal y limpio.

El flujo con rebase es:

  1. Volvemos a main y hacemos pull para tener lo último
  2. Volvemos a nuestra rama
  3. Hacemos rebase de main, que pone nuestros commits encima
  4. Historial limpio, como si hubiéramos empezado a trabajar desde el último commit de main

Visualmente

Merge:
main:    A---B---C---M
              \     /
feature:       D---E

Rebase:
main:    A---B---C
                  \
feature:           D'---E'

Comandos

# Crear una rama de trabajo
git co -b feature/contact-page

Código: página de contacto

Crear resources/views/contact.blade.php:

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Contact - Git Pro Course</title>
</head>
<body>
    <h1>Contact</h1>
    <p>Email us at hello@gitpro.test</p>
</body>
</html>

Añadir ruta en routes/web.php:

Route::get('/contact', function () {
    return view('contact');
});

Comandos

git add .
git cm "feat: add contact page"

# Simulamos que main ha avanzado.
# En la vida real, esto vendría de un git pull tras los cambios de un compañero.
# Aquí lo hacemos directamente en main para simular esa situación.
git co main

Código: cambio en main mientras trabajábamos

Editar resources/views/about.blade.php, añadir versión:

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>About - Git Pro Course</title>
</head>
<body>
    <h1>About</h1>
    <p>This project is built with Laravel and Sail.</p>
    <p>Version 1.0</p>
</body>
</html>

Comandos

git add .
git cm "docs: add version info to about page"

# Ver el historial actual (las ramas han divergido)
git lg

# Flujo completo de rebase:
# 1. Actualizar main con lo último del remoto
git co main
git pull

# 2. Volver a nuestra rama
git co feature/contact-page

# 3. Rebase: poner nuestros commits encima de main actualizado
git rebase main

# Ver el historial limpio y lineal
git lg

# Integrar en main
git co main
git merge feature/contact-page

# El historial queda completamente lineal (fast-forward)
git lg

# Subir y limpiar
git push
git br -d feature/contact-page

⚠️ ¿Qué pasa si hay conflictos durante el rebase?

Cuando Git no puede aplicar tus commits automáticamente (porque hay cambios en las mismas líneas), el rebase se pausa y te pide resolver el conflicto manualmente.

Ejemplo de conflicto

# Durante el rebase aparece:
# CONFLICT (content): Merge conflict in routes/web.php

# Ver qué archivos tienen conflictos
git status

# Abrir el archivo y verás marcadores:
# <<<<<<< HEAD (cambios de main)
# código de main
# =======
# código de tu rama
# >>>>>>> tu-commit

# Editar el archivo, decidir qué código conservar, eliminar los marcadores

# Una vez resuelto
git add routes/web.php
git rebase --continue
# ⚠️ NO hagas git commit (error común)
# git rebase --continue se encarga de eso automáticamente

# Si te pierdes y quieres cancelar todo
git rebase --abort

Diferencia con merge: en merge resuelves el conflicto una sola vez y haces git commit. En rebase NO haces commit, solo git add y git rebase --continue.

Lección 09

Rebase Interactivo - Squash, Fixup y Reorder

El rebase interactivo es una de las herramientas más potentes de Git. Nos permite reescribir el historial de commits antes de integrar nuestro trabajo: podemos unir varios commits en uno (squash), eliminar commits, reordenarlos o editar sus mensajes.

Esto es muy útil cuando estamos trabajando en una feature y vamos haciendo commits pequeños tipo "wip", "fix typo", "ahora sí funciona"... Antes de hacer merge a main, limpiamos todo eso y dejamos commits limpios y profesionales.

Los comandos del rebase interactivo

  • pick: mantener el commit tal cual
  • reword: cambiar el mensaje del commit
  • squash: unir con el commit anterior (mantiene ambos mensajes)
  • fixup: unir con el commit anterior (descarta este mensaje)
  • drop: eliminar el commit

Comandos

# Crear una nueva rama
git co -b feature/services-page

Código: estructura inicial

Crear resources/views/services.blade.php:

<!DOCTYPE html>

    <title>Services - Git Pro Course</title>
</head>
<body>
    <h1>Services</h1>
</body>
</html>

Comandos

git add .
git cm "feat: add services page"

Código: añadir listado

Editar resources/views/services.blade.php:

        <li>Web Development</li>

Añadir en routes/web.php:

Código: corregir título

Editar resources/views/services.blade.php, cambiar el título:

    <h1>Our Services</h1>

Comandos

git add .
git cm "fix: update services page title"

# Ver los 4 commits que tenemos
git lg

# Limpiar: queremos dejar un solo commit limpio
# Rebase interactivo de los últimos 4 commits
git rebase -i HEAD~4

En el editor que se abre

reword abc1234 feat: add services page
fixup abc1235 feat: add services list
fixup abc1236 feat: add services route
fixup abc1237 fix: update services page title

Cambiar pick por reword en el primero y por fixup en los tres últimos. Guardar y cerrar. Se abrirá otro editor para cambiar el mensaje del primer commit a: feat: add services page with route and listing.

git pull
git co feature/services-page
git rebase main
git co main
git merge feature/services-page
git push
git br -d feature/services-page
Lección 10

Cherry-pick - Eligiendo Commits Específicos

Cherry-pick nos permite coger un commit concreto de cualquier rama y aplicarlo en nuestra rama actual. Es como decir "quiero solo este cambio, no toda la rama".

Casos de uso reales: un compañero ha hecho un fix en su rama que necesitamos ya en main sin esperar a que termine toda su feature. O hemos hecho un commit en la rama equivocada y queremos moverlo.

Comandos

# Crear una rama simulando el trabajo de un compañero
git co -b feature/improvements

Código: commit que NO queremos traer

Editar resources/views/home.blade.php, añadir un subtítulo debajo del h1:

    <h1>Welcome to Git Pro Course</h1>
    <h2>Work in progress</h2>

Comandos

git add .
git cm "feat: add subtitle to home page"

Código: commit que SÍ queremos traer

Editar resources/views/about.blade.php, añadir licencia:

    <p>Version 1.0</p>
    <p>Licensed under MIT.</p>

Comandos

git add .
git cm "docs: add license info to about page"

# Ver los commits y copiar el hash del que queremos
git lg

# Volver a main y traer solo el commit que nos interesa
git co main
git cherry-pick <HASH_DEL_COMMIT_DE_LICENSE>

# Verificar que solo tenemos ese cambio
git lg
git push

# Limpiar la rama (usamos -D en vez de -d porque tiene commits sin mergear)
git br -D feature/improvements
Lección 11

Git Bisect - Encontrando el Commit que Rompió Todo

Imaginemos que algo ha dejado de funcionar y no sabemos cuándo se rompió. Tenemos 50 commits por delante. ¿Revisamos uno por uno? No. Git bisect hace una búsqueda binaria automática: le decimos cuál es un commit bueno (donde todo funcionaba) y cuál es malo (donde está roto), y Git va partiendo por la mitad hasta encontrar el commit exacto que introdujo el problema.

Es una herramienta de depuración increíblemente útil que poca gente conoce.

Preparación: creamos varios commits para simular un historial real

Commit 1: actualizar navegación

Editar resources/views/home.blade.php, actualizar nav:

    <nav>
        <a href="/">Home</a>
        <a href="/about">About</a>
        <a href="/services">Services</a>
        <a href="/contact">Contact</a>
    </nav>
git add .
git cm "feat: update navigation with all links"

Commit 2: el que introduce el error

Editar routes/web.php, comentar la ruta de contact (simulando un error):

// Route::get('/contact', function () {
//     return view('contact');
// });
git add .
git cm "refactor: reorganize routes file"

Commit 3: cambio en services

Editar resources/views/services.blade.php, añadir párrafo antes del cierre de body: 

    <p>Contact us for custom solutions.</p>
</body>
git add .
git cm "feat: add call to action in services page"

Commit 4: cambio en about

Editar resources/views/about.blade.php, añadir línea antes del cierre de body:

    <p>Built with love and Laravel.</p>
</body>
git add .
git cm "docs: add tagline to about page"
git push

Usando bisect para encontrar el problema

# Ahora "descubrimos" que /contact no funciona

# Iniciar bisect
git bisect start

# Marcar el commit actual como malo
git bisect bad

# Marcar un commit anterior donde sabemos que funcionaba
git bisect good HEAD~4

# Git nos posiciona en un commit intermedio
# Comprobamos si la ruta /contact está registrada
sail artisan route:list --path=contact

# Si no aparece la ruta -> git bisect bad
# Si aparece -> git bisect good
# Repetir hasta que Git nos diga cuál fue el primer commit malo

# Terminar bisect y volver al estado normal
git bisect reset
Lección 12

Git Reflog - Recuperando lo Perdido

Reflog es el registro de todo lo que ha pasado con HEAD en nuestro repositorio. Cada vez que hacemos checkout, commit, rebase, reset... queda registrado. Es nuestra red de seguridad: si hacemos un reset --hard, un rebase que sale mal, o borramos una rama sin querer, reflog nos permite volver atrás.

Git guarda este registro durante 90 días por defecto. Es local, no se comparte con el remoto.

Comandos

# Ver el reflog actual
git reflog

# Vamos a simular un "desastre": borrar los últimos 2 commits
git lg
git reset --hard HEAD~2

# Hemos perdido commits
git lg

# Pero reflog lo tiene todo
git reflog

# Recuperar volviendo al commit que queremos
git reset --hard <HASH_DEL_REFLOG>

# Verificar que hemos recuperado todo
git lg

# Otro caso: recuperar una rama borrada
git co -b feature/temp
git co main
git br -D feature/temp

# La rama ya no existe
git br

# Pero podemos ver en reflog cuándo estuvimos en ella
git reflog

# Recrear la rama desde el reflog
git co -b feature/temp <HASH_DEL_REFLOG>
git co main
git br -D feature/temp
Lección 13

Estrategias de Branching y Convenciones

No hay una única forma correcta de trabajar con ramas, pero hay dos estrategias dominantes:

Git Flow

Ramas de larga vida (main, develop, feature/, release/, hotfix/). Cada tipo tiene su propósito. Es más estructurado, ideal para proyectos con ciclos de release definidos. 

main ─────────────────────────── (producción)
  │
  └── develop ────────────────── (desarrollo)
       │         │
       └── feature/login    └── feature/cart

Trunk Based Development

Una sola rama principal (main/trunk). Las features se hacen en ramas muy cortas (1-2 días máximo) y se integran rápidamente. Es más ágil, ideal para equipos con CI/CD. 

main ──●──●──●──●──●──●──●──── (todo va aquí)
        \  /    \  /
         \/      \/
       (ramas cortas)

En este curso usamos Trunk Based Development: trabajamos directamente con main y ramas feature cortas. Es el flujo más habitual en equipos pequeños y medianos, y el más sencillo de mantener. Creamos una rama, trabajamos, hacemos rebase con main para actualizar, e integramos.

Independientemente de la estrategia, las ramas deben seguir convenciones de nombre claras:

  • feature/nombre — nueva funcionalidad
  • fix/nombre — corrección de un bug
  • chore/nombre — tareas de mantenimiento

El nombre debe ser corto y descriptivo: feature/faq-page, fix/broken-contact-route, chore/update-dependencies. Así cualquiera sabe de un vistazo qué hace esa rama.

Lección 14

Pull Requests - Revisión de Código Profesional

En el curso estamos integrando directamente en local con merge porque es lo más directo. Pero en equipos donde hay revisión de código, en lugar de hacer merge en local, subimos nuestra rama a GitHub y creamos un Pull Request (PR). El PR permite que otros revisen nuestro código antes de integrarlo en main. GitHub se encarga del merge.

El flujo cambia solo en el último paso: en vez de merge en local, subimos la rama y creamos PR. Todo lo anterior (crear rama, trabajar, rebase) es exactamente igual.

Un buen PR tiene un título descriptivo (con Conventional Commits), una descripción que explique qué se ha hecho y por qué, y cambios atómicos (una feature o fix por PR).

Buenas prácticas

  • PRs pequeños y enfocados (fáciles de revisar)
  • Título con Conventional Commits: feat: add user authentication
  • Descripción clara de qué y por qué
  • Solicitar revisión a compañeros
  • No mergear tu propio PR si hay revisores disponibles
  • Responder a los comentarios antes de mergear

Sobre el botón de merge en GitHub: veremos tres opciones. "Create a merge commit" añade un commit de fusión. "Squash and merge" aplasta todos los commits de la rama en uno solo, útil si no hemos limpiado los commits antes. "Rebase and merge" reaplica los commits encima de main. Si ya hemos hecho squash con rebase interactivo en local (como vimos en la lección 9), cualquier opción funciona igual porque solo tenemos un commit limpio. Si no hemos limpiado, "Squash and merge" es la opción más cómoda.

Comandos

git co -b feature/improve-navigation

# Mismo flujo de siempre hasta el rebase

# En vez de merge local, subimos la rama a GitHub para crear el PR
git push -u origin feature/improve-navigation

Pasos en GitHub

  1. Ir al repositorio > Pull Requests > New Pull Request
  2. Seleccionar feature/improve-navigation → main
  3. Título: feat: improve navigation with all page links
  4. Descripción:
## What
Updated the navigation menu on the home page to include links to all available pages.

## Why
Users couldn't navigate to FAQ and Services from the home page.
  1. Revisar la pestaña "Files changed" (revisión de código)
  2. Añadir comentarios en líneas específicas si es necesario
  3. Aprobar y mergear (recomendado: Squash and Merge)

Después del merge en GitHub

git co main
git pull
git br -d feature/improve-navigation
git push origin --delete feature/improve-navigation
Lección 15

Buenas Prácticas y Flujo de Trabajo Profesional

Resumen de todo lo aprendido aplicado al flujo que hemos usado durante todo el curso:

Flujo de trabajo completo

# 1. Actualizar main
git co main
git pull

# 2. Crear rama
git co -b feature/nueva-funcionalidad

# 3. Trabajar con commits atómicos
git st
git diff
git add .
git ds
git cm "feat: description"

# 4. Limpiar commits si hace falta
git rebase -i HEAD~2

# 5. Actualizar main y rebase
git co main
git pull
git co feature/nueva-funcionalidad
git rebase main

# 6. Integrar en main
git co main
git merge feature/nueva-funcionalidad
git push

# 7. Limpiar
git br -d feature/nueva-funcionalidad

Si trabajamos en un equipo con revisión de código, el paso 6 cambia: en vez de merge en local, subimos la rama con git push -u origin feature/nueva-funcionalidad y creamos un PR en GitHub, como vimos en la lección 14.
